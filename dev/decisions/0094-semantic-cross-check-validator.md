# 0094 — The semantic cross-check validator (the second validation layer): scope, naming, routing seam, two doc gaps

**Date:** 2026-06-17
**Phase:** 2 (Task 5 — Validator Layer 2, cross-block-within-manifest)
**Architecture refs:** `validation.md §6.5` (semantic cross-check — what it checks, read-only/findings-not-mutation), `§6.10`/`§6.10.2` (findings route to the responsible agent; the orchestrator owns the stack), `architecture.md §1.6` (mechanical, never LLM; read-only). Subclasses the ADR-0073 `Finding`/`FindingResult` base; locators per ADR 0074. Schema cross-refs: `schema.md §4.13` (facet `implies`/`incompatible_with`), `§4.5` (`identifier_source` resolution), `§4.4` (`affected_platforms` derived from facets).

## Context

Task 5 builds the second mechanical validation layer (`validation.md §6.5`, "semantic cross-check") over the `LabManifest`. In Phase 2 the **cross-block-within-manifest** checks are live; the **code-vs-manifest** checks (which need generated IaC/code) are built but **inert** until Phase 3. The layer **flags findings; it never mutates the manifest** (`§6.5`, `§1.6`).

A pre-work verification pass (3-agent fan-out + adversarial refutation, against real `HEAD`) resolved three gates before coding: the `affected_platforms` check is **vacuous** in the locked v1 schema (decision 4); the `identifier_source` form is **unconstrained by the model but documented** as `phase_outputs.<name>` (decision 5); and the Task-4 refinement resolver had an unrelated regression, fixed first (ADR 0091 Amendment).

## Decision

1. **Naming (no ordinal token).** Module `validators/semantic_cross_check_validator.py`; `SemanticCrossCheckCode` / `SemanticCrossCheckFinding` / `SemanticCrossCheckResult` / `SemanticCrossCheckValidator`. This is the `§6.5` architectural title ("semantic cross-check"); the token "layer 2"/"L2" appears in no identifier, per ADR 0046 and the now-general `coding-conventions.md §5.5`. The brief's `L2Code` is a placeholder resolved to the descriptive name here.

2. **Live cross-block checks** (`§6.5`):
   - **Facet `implies`** — for each declared facet, every facet in its registry entry's `implies` must also be declared; a missing one is a `MISSING_IMPLIED_FACET` finding. Direct (one level), matching the `§4.13`/`§6.5` documented behaviour ("declare `target:eks` → confirm `target:aws` and `target:kubernetes`"). Never auto-added — the Validator stays read-only; the Planner adds it next iteration.
   - **Facet `incompatible_with`** — a declared pair where either side lists the other is an `INCOMPATIBLE_FACETS` finding (symmetric; each unordered pair reported once).
   - **`produces_world_state` identifier_source resolution** — for each phase, each `produces_world_state` entry with `identifier_kind: runtime_generated` must carry an `identifier_source` of the canonical form `phase_outputs.<name>` resolving to a declared `outputs[].name` on that phase; otherwise `UNRESOLVED_IDENTIFIER_SOURCE`.

3. **Inert code-vs-manifest checks (Phase-3, built + marked).** The `references_lab_outputs` bidirectional cross-check (`§6.5`) needs generated IaC, which does not exist until Phase 3. Its two finding codes are reserved — `UNDECLARED_LAB_OUTPUT_REFERENCE` (per-phase IaC references a lab-level output the Lab-level Generator didn't produce) and `UNDECLARED_LAB_RESOURCE_REFERENCE` (per-phase IaC references a `lab_resources` entry the Planner didn't declare) — and a clearly-marked `TODO(phase-3)` method documents both directions and returns no findings. Phase 3 supplies the generated-IaC reference set and wires it; the codes and the routing already exist so Phase 3 does not re-derive them.

4. **The `affected_platforms` consistency check is VACUOUS in v1 — not implemented; surfaced to the architect.** `§6.5` says Layer 2 verifies a `core.affected_platforms` field against the `target:*` facets "if the manifest's core block has an `affected_platforms` field (where present in user-edited manifests)." But `CoreBlock` (the locked Task-1 schema) declares **no** `affected_platforms` field and is `extra="forbid"`, and `§4.4` states affected platforms "are not duplicated here — they're derivable from the `facets` block's `target:*` entries." So a Layer-1-valid manifest can never carry the field; the check has no left-hand operand and can never fire. The code `INCONSISTENT_AFFECTED_PLATFORMS` is **reserved** (documented) but no check is implemented — implementing one would be dead code, and adding the field would violate the manifest lock and `§4.4`. This is a `validation.md §6.5` ↔ `schema.md §4.4` drift: `§6.5` anticipates a field the v1 schema forbids. **Surfaced for the architect** to reconcile (`§6.5` should note the check is v1-vacuous, or a future schema bump may add the field with the Planner persisting it). Not silently resolved.

5. **`identifier_source` canonical form.** The `ProducesWorldState.identifier_source` field is a free `NonEmptyString | None` — the Pydantic model imposes no format. The canonical form is documented: `§4.5`'s example (`identifier_source: phase_outputs.malicious_branch_name`) and `§6.5` ("a path like `phase_outputs.malicious_branch_name` must correspond to a key in the phase's declared `outputs` block"). Layer 2 resolves the segment after a `phase_outputs.` prefix against the phase's `outputs[].name`; a value that lacks the prefix or names an undeclared output is flagged. `§4.5`'s parenthetical "or another known output location" is not modeled in v1 (no other location exists in the schema); enforcing the documented `phase_outputs.` form does not contradict the model (which is format-free) or the docs.

6. **Routing seam — the validator never routes; a pure mapping does.** Per the `validators/` contract (each layer *returns findings*; the orchestrator routes), the validator class has no routing. A module-level `responsible_agent_for(finding) -> ResponsibleAgent` maps every **live** Phase-2 finding code to `ResponsibleAgent.PLANNER` — in Phase 2 the Planner is the only implementation agent that exists, and all live cross-block findings are manifest-declaration issues it owns (`§6.5`/`§6.10`). This is the seam **Task 6** consumes when it inserts the semantic-cross-check node into the plan graph and wires the route-to-Planner edge. The reserved Phase-3 codes have no Phase-2 route (they are inert) and the mapping raises for them.

7. **Graph-node insertion is Task 6, not Task 5.** The brief's Task-5 item 4 ("wire Layer 2 into the orchestrator-owned mechanical stack; findings route to the Planner") overlaps Task-6 item 2 ("wire the linear graph: Planner → Planner-Jury → Layer 2"), and the Task-4 execution log assigned the graph-node insertion to Task 6. Resolved conservatively: Task 5 delivers the validator + the `responsible_agent_for` routing contract (unit-tested → PLANNER), satisfying the "routes to the Planner" requirement as a contract; Task 6 inserts the node into the `plan` graph and connects the edge, consuming the same mapping (one mechanism, no divergence). Task-5 exit criteria are validator-scoped and do not require the graph node.

8. **The non-first-class `runtime:*` warning (`§6.5`) is deferred.** `§6.5` says Layer 2 emits a *warning* (not a failure) when the manifest declares a non-first-class runtime. It is **not** in Task 5's work-items, and — the real reason — the ADR-0073 `Finding` base carries no **severity / warning level** (`code`, `location`, `detail` only), so a warn-level finding is currently inexpressible. Bolting a severity onto the shared base for one out-of-scope warning is the wrong trade. Deferred with that gap recorded; whoever needs warn-level findings adds severity to the base deliberately.

## Consequences

- New `validators/semantic_cross_check_validator.py` (validator + codes + finding/result + the routing mapping); re-exported from `validators/__init__.py`.
- The live checks fire on crafted manifests; the validator is read-only (a `model_dump` before/after equality test pins it); the inert references-lab-outputs method exists and is a no-op (pinned); `responsible_agent_for` returns PLANNER for each live code (pinned).
- **No `docs/` (architecture-tier) edit in this task.** The `affected_platforms` drift (decision 4) and the deferred runtime warning (decision 8) are surfaced here and in the execution log for the architect; not silently edited into `validation.md`. The naming rule landed separately in `coding-conventions.md §5.5`.
- Task 6 wires the node + persistence; Phase 3 lights up the inert checks and (if needed) adds finding severity for the runtime warning.

## Alternatives considered

- **Implement `affected_platforms` defensively anyway.** Rejected: it is unreachable dead code (the field cannot exist on a Layer-1-valid manifest); the honest action is to reserve the code and surface the doc drift.
- **Insert the Layer-2 node into the plan graph here.** Rejected: duplicates/contends with Task-6 item 2 and the Task-4 deferral; Task 5 ships the routing contract Task 6 consumes instead.
- **Add a `severity` field to the `Finding` base now for the runtime warning.** Rejected: out of Task-5 scope, and a one-off reason to change a shared contract; deferred with the gap named.
- **A bespoke finding/result pair instead of subclassing ADR 0073.** Rejected by the brief's "no discretion" — subclass the shared base.
